# 项目概述

`ComfyUI-Notifier` 是一个 ComfyUI 通知节点扩展，用于把工作流产物推送到 Telegram、Discord、Yike 等外部渠道。当前版本在保持原有 `file_path` 输入兼容的前提下，新增了原生 `IMAGE`、`AUDIO`、视频路径统一适配，以及面向未来渠道扩展的“能力声明 + 投递策略解析 + 自动降级”架构。

# 快速开始

## 安装

1. 将目录放到 [`custom_nodes/ComfyUI-Notifier`](custom_nodes/ComfyUI-Notifier) 。
2. 安装依赖：[`custom_nodes/ComfyUI-Notifier/requirements.txt`](custom_nodes/ComfyUI-Notifier/requirements.txt)
3. 配置 `.env`：Telegram / Discord / Yike 对应令牌。
4. 重启 ComfyUI。

## 运行

在工作流中使用 [`GeneralNotifier`](custom_nodes/ComfyUI-Notifier/nodes.py:4) 节点：

- 旧用法：仅传入 `file_path`
- 新用法：可直接传入 `image`、`audio`、`video`
- `delivery_mode` 支持 `auto / media / file / zip`
- `execution_mode` 支持 `async / sync`，默认 `async`
- `parallel_dispatch` 控制多渠道并行或串行，默认 `True`
- `retry_attempts` 失败重试次数，默认 `0`（不重试）
- `retry_delay_seconds` 首次重试等待秒数，默认 `1.5`
- `retry_backoff_factor` 指数退避倍率，默认 `2.0`
- 为兼容旧工作流仍保留 `send_as_file`、`send_as_zip`，它们会覆盖 `delivery_mode`

# 文件说明

## 核心设计

### 统一媒体抽象

[`MediaAdapter.resolve_payload()`](custom_nodes/ComfyUI-Notifier/notifier/media_adapter.py:58) 会把传统路径、ComfyUI 原生 `IMAGE`、原生 `AUDIO`、视频路径统一转换为 [`MediaPayload`](custom_nodes/ComfyUI-Notifier/notifier/media_adapter.py:31)。

### 渠道能力声明

每个渠道通过 [`Notifier.capabilities()`](custom_nodes/ComfyUI-Notifier/notifier/base.py:55) 声明自己对以下能力的支持情况：

- `media`
- `file`
- `zip`
- 媒体类别支持：`image / audio / video / binary`

这让新增渠道时不需要在调度器里硬编码判断，只需在新渠道类中提供能力矩阵。

### 投递策略解析与统一降级

[`Notifier.resolve_delivery_plan()`](custom_nodes/ComfyUI-Notifier/notifier/base.py:62) 根据用户请求和渠道能力生成 [`DeliveryPlan`](custom_nodes/ComfyUI-Notifier/notifier/base.py:24)：

- 用户请求 `auto` 时，优先 `media`，不支持则降级到 `file`，再降级到 `zip`
- 用户请求特定模式时，如果渠道不支持，会生成带降级原因的回退计划
- 若所有模式都不支持，则抛出明确错误

### 执行模型

当前支持两种执行模式：

- `async`：后台异步派发，默认值，不阻塞后续 ComfyUI 节点
- `sync`：当前节点等待渠道发送完成后再返回

同时支持两种分发方式：

- `parallel_dispatch=True`：多个渠道并行发送
- `parallel_dispatch=False`：多个渠道串行发送

默认组合是“异步 + 并行”，用于获得最高吞吐。

### 失败重试策略

当前新增统一失败重试能力，适用于 Telegram / Discord / Yike：

- 默认 `retry_attempts=0`，保持历史行为（失败即返回，不重试）
- 仅在可重试异常（连接失败、超时、SSL/EOF、协议层临时错误等）时触发重试
- 使用指数退避：`retry_delay_seconds * retry_backoff_factor^n`
- 每次发送日志会记录 `attempt=当前/总次数`，方便定位是否为首发失败或重试失败

说明：日志中 `Max retries exceeded` 可能来自底层 HTTP 库连接池机制，不等于节点业务层已重试；节点业务层重试由上述参数控制。

### 结构化结果

[`DeliveryResult`](custom_nodes/ComfyUI-Notifier/notifier/base.py:40) 和 [`NotificationResult`](custom_nodes/ComfyUI-Notifier/notifier/notify.py:13) 会记录：

- 请求模式
- 实际执行模式
- 是否降级
- 降级原因
- 发送状态
- 执行耗时（毫秒）

这为后续 UI 展示、日志分析、失败重试提供了基础。

### 缓存与性能

- 对 `AUDIO` / `IMAGE` 物化结果按内容哈希缓存，避免重复导出
- 默认使用后台线程异步派发，不阻塞后续 node
- 阻塞型 HTTP / webhook 调用通过 [`Notifier.run_blocking()`](custom_nodes/ComfyUI-Notifier/notifier/base.py:107) 放入线程池执行
- zip 发送统一使用内存压缩，减少磁盘 IO 与临时文件残留
- 日志会记录请求进入、队列投递、渠道解析、成功/失败、耗时、后台分发开始与结束

## 渠道策略说明

### Telegram

- `media`：优先走图片 / 音频 / 视频专用接口
- `file`：统一走 `sendDocument`
- `zip`：压缩后作为文档发送
- 已补充 `connect/read/write/pool` 超时参数，降低长连接抖动下的偶发失败概率

### Discord

- `media`：附件上传 + embed 呈现
- `file`：附件上传，不额外构造媒体型 embed
- `zip`：zip 附件上传

### Yike

- 不支持真正的 `media` 消息语义
- `auto/media` 会自动降级到 `file`
- `zip` 可直接上传压缩包

# 环境要求

- Python 版本需与当前 ComfyUI 运行环境一致
- 建议安装 Pillow、PyAV、torch（通常随 ComfyUI 环境已有）
- 外部通知渠道需要各自合法可用的凭据

# 注意事项

- AI 已完成架构升级和编译校验，但我无法在你的真实第三方凭据环境中做端到端发送验证，需要你自行实测
- `IMAGE` 输入会物化为 PNG，`AUDIO` 输入会物化为 FLAC/指定扩展对应格式；这是为了兼容 ComfyUI 内部对象
- 如果未来新增渠道，只需实现新的 [`Notifier`](custom_nodes/ComfyUI-Notifier/notifier/base.py:50) 子类，并声明能力矩阵，无需修改节点层与媒体适配层
